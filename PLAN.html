🚀 Antigravity IDE: Granular YOLO Extension

Objective: Implement a tiered execution engine within the Antigravity IDE environment. Instead of a binary YOLO toggle, the extension will route the agent's commands through a permission matrix, allowing autonomous execution for safe operations while enforcing manual prompts for everything else.

1. Execution Tiers (The Permission Engine)

Level 0: Interactive Default

Behavior: Standard operation. Every system command generated by the agent halts execution and requires explicit user confirmation via the Antigravity notification or terminal prompt.

Level 1: Read-Only YOLO Safe Autonomy

Behavior: Auto-executes non-destructive discovery commands so the agent can build its own project context without nagging the user.

Whitelist Examples: cat, ls, pwd, grep, git status, git diff.

Level 2: Scoped YOLO Targeted Execution

Behavior: Allows the agent to automatically execute specific operational types based on user configuration. Perfect for letting the agent handle a localized workflow (like debugging tests) autonomously.

  • Package Ops: npm install, pip install
  • Build/Test Ops: tsc, npm test, pytest
  • Git Ops: git commit, git push

Note: If a command falls outside the active scope, it downgrades to Level 0 and prompts the user.

Level 3: Full YOLO Unrestricted

Behavior: The agent has full, unsupervised execution rights on the host machine. Bypasses all confirmation prompts.

Hard Failsafes: Even in Level 3, a strict blacklist will immediately intercept catastrophic commands (e.g., rm -rf /, mkfs) and kill the recursive execution loop.

2. Extension Architecture in Antigravity

  • Command Interceptor: Hook into the Antigravity agent's tool-call execution cycle. Before the agent pipes a command to the integrated terminal, intercept and parse the command string.
  • Regex Parsing & Tokenization: Split the command into the primary executable (e.g., npm) and arguments (e.g., install). Match these tokens against the user's configured execution level.
  • Configuration UI: Utilize the standard package.json configuration contributions to add a settings panel in the IDE, allowing users to define their active YOLO level and custom command whitelists for Level 2.

3. Implementation Roadmap

  • Phase 1: Settings & UI. Register the extension configurations to expose the dropdown for YOLO Levels (0-3) and array inputs for custom whitelists.
  • Phase 2: The Filter Pipeline. Build the CommandValidator class that accepts a shell string and the current YOLO tier, returning an object state: { execute: boolean, promptRequired: boolean }.
  • Phase 3: Antigravity Integration. Connect the validator to the child process spawning the command. If promptRequired is true, surface the IDE confirmation dialog.
  • Phase 4: Feedback Loop. Stream the stdout and stderr of auto-executed commands back into the agent's context window so it can read its own outputs and iterate.
